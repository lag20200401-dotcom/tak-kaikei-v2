## 1. システム概要

### システム名
整体院TAK経営管理システム v2(tak-kaikei-v2)

### 目的
整体院TAKの日々の売上・経費・顧客管理を行い、月次でExcel形式の業績資料を出力する経営管理システム。

### アーキテクチャ
- フロントエンド: HTML + Alpine.js + Tailwind CSS(vanilla JS)
- ホスティング: GitHub Pages
- データベース: Firebase Firestore(Sparkプラン・無料枠)
- バックアップ: Google Sheets「TAK会計_クラウド」連携
- 月次出力: SheetJS + JSZip による3ファイルZIP出力

### 運用形態
- メイン操作: Tak さん(オーナー・ソロ運用)
- サブ操作: 日向さん(既存顧客のインディバ施術記録のみ)
- 月額コスト: ¥0(Firebase Sparkプラン継続)

### 利用デバイス
- Tak さん: Mac(編集・閲覧) + Pixel 8(現場入力)
- 日向さん: 任意のデバイス(既存顧客の visits / sales 入力のみ)

## 3. 日々の運用ルール

### 朝のルーティン(業務開始前)

1. **Firebase Console「使用状況」タブを開いて読取数チェック**
   - 過去24時間の読取数が異常に多くないか(通常 5,000〜10,000件以内)
   - 20,000件超えていたら要注意、原因調査
   - URL: https://console.firebase.google.com/project/tak-kaikei-v2/firestore/databases/-default-/usage/last-60m

2. **アプリ起動 → ロック解除 → データ表示確認**
   - パスコード `1111` でアンロック
   - 顧客一覧・最近の visits / sales が正常表示されるか確認
   - 異常があれば「7. トラブル対応手順」参照

### 日中の入力作業

1. **アプリタブは1つだけ開く**(最重要ルール、後述の「8. やってはいけないこと」参照)
2. 来院記録は visits タブで都度入力
3. 売上は sales タブで都度入力
4. 物販は sales タブの該当区分で入力
5. 新規顧客が来たら customers タブで登録(Tak さん専任)

### 夜のルーティン(業務終了時)

1. **20:00時点で Firebase Console 読取数チェック**
   - 40,000件超えていたら作業停止して原因調査
   - 通常運用なら 10,000〜20,000件程度に収まる想定

2. **当日入力データの確認**
   - その日の visits / sales が正しく入力されているか
   - 抜け漏れがあれば追加入力

3. **アプリタブを閉じる**
   - リスナーを解放してFirestore接続を切る
   - 翌朝の初回起動でキャッシュから顧客件数取得

### 週次ルーティン(任意・推奨)

- **日曜夜**: 当週の入力データ確認、Firebase使用状況の傾向チェック
- **月曜朝**: 前週の visits / sales / expenses / ads が完全に入力されているか確認

### 月次ルーティン

- **月初1〜3日**: 前月分の月次Excel出力(3ファイルZIP)
- **月初**: 出力したExcelを日向さんへ提出
- **月初**: 月次バックアップ(後述「5. バックアップ運用」参照)
- **毎月1日**: サブスクタブで前月分の支払い状況確認・当月分の月次生成(v0.9〜)

- visits / sales / products / expenses / ads / cash_deposits / purchases / subscriptions / subscription_payments は **onSnapshotでリアルタイム同期**
- どちらか一方が更新すれば、もう片方の端末でも数秒以内に反映される
- 回数券残数も sales がリアルタイムなので双方で正確に表示される

### 例外: 新規顧客が必要になった場合

1. 日向さん施術中に新規飛び込み発生
2. 日向さんがLINEで Tak さんに連絡(顧客名・電話番号・生年月日)

4. 日向さんの端末で以下のいずれかを実施:
   - 推奨: ブラウザConsoleで `app().refreshCustomersTemp()` を実行
   - もしくは: `localStorage.removeItem('tak_customersTemp_cache_v1')` 実行 → リロード
5. 新規顧客が反映されたことを確認してから visits / sales 入力

### 日向さん用の最低限ルール

1. **アプリタブは1つだけ開く**
2. customers タブ・サブスクタブは触らない
3. 新規顧客登録が必要な場面では Tak さんに連絡
4. 入力ミスに気づいたら速やかに修正、または Tak さんに連絡
5. 不明点があれば Tak さんに確認してから操作

## 7. トラブル対応手順

### 症状A: アプリが起動しない・データが表示されない

1. ブラウザのリロード(Shift + リロードで強制リロード)
2. それでも駄目なら Console(F12)でエラー確認
3. Firebase Console で読取数が上限到達していないか確認
4. localStorage破損の疑いがあれば以下実行:
   ```
   localStorage.removeItem('tak_customersTemp_cache_v1')
   ```
   その後リロード

### 症状A': アプリが真っ白(コミット直後)

→ 「12. GitHub Web Editor 編集時の注意事項」参照
1. GitHub の Commits 履歴から1つ前のコミットを開く
2. そのファイルの内容を main ブランチに復元してCommit(緊急時のロールバック)
3. SyntaxError の行番号を Console から確認 → 該当箇所を修復

### 症状B: 顧客一覧に追加したはずの新規顧客が表示されない

1. 別端末で追加された場合: localStorage キャッシュ24h経過待ちが必要
2. 即時反映したい場合:
   ```
   app().refreshCustomersTemp()
   ```
   または
   ```
   localStorage.removeItem('tak_customersTemp_cache_v1')
   ```
   実行後リロード

### 症状C: Firestore読取数が異常に多い

1. アプリのタブを複数開いていないか確認
2. Firebase Console を別タブで開きっぱなしになっていないか確認
3. 大量CSV取込を直前に実行した場合は一時的にクォータ消費する(許容)
4. 原因不明で40,000件超 → 作業停止して翌日に持ち越し

### 症状D: 月次Excel出力でエラー

1. ブラウザConsole(F12)でエラー内容確認
2. SheetJS/JSZipの読み込み失敗の可能性 → ネット接続確認
3. データ件数異常の可能性 → 該当月のvisits/sales件数を確認
4. 解決しない場合は前月の出力履歴をmonthlyExportsから確認

### 症状E: 日向さんの端末で同期が遅い・反映されない

1. 日向さんの端末でブラウザをリロード
2. それでも駄目ならアプリのタブを一度閉じて再起動
3. ネット接続確認
4. Firestore側の障害可能性 → https://status.firebase.google.com 確認

### 症状F: GitHub Pagesで更新が反映されない

1. Shift + リロードまたはシークレットモードで確認
2. GitHub Pagesのビルド完了まで数分待つ
3. GitHub Actions のビルドログでエラー確認

### 症状G: GAS deploy URL の問題

- 過去事故: lowercase `il` を `IL` に間違えて URL が機能しない事案あり
- 正しいURL: `AKfycbzXlf7EYbywxJG7EVGOZtpLUWS5UCIL4rFCCqtgRY2tOaQdcgKryBK4sXZFwO0MWJ1S/exec`
- 必ず Claude のチャット履歴からコピー(GAS UI からのコピーは全角クォート混入の事例あり)

### 症状H: サブスク月次生成ボタンを押しても何も起きない(v0.9〜)

1. Console で以下を実行して関数が存在するか確認:
   ```
   typeof Alpine.$data(document.querySelector('[x-data]')).generateMonthlyPayments
   ```
   → `'function'` が返らない場合 → JavaScript部分が壊れている可能性 → コミット履歴確認
2. writeBatch が import されているか確認:
   ```
   typeof window._fb.writeBatch
   ```
   → `'function'` が返らない場合 → コミット1が反映されていない
3. アクティブなスクエア契約が0件の場合 → 契約マスター追加から開始

## 8. やってはいけないこと

### 🚨 絶対禁止

1. **同じアプリのタブを複数開く**
   - 過去にFirestoreクォータ超過の主因となった
   - リスナーが端末ごとに重複起動して読取数が倍増する

2. **本番DBで実験操作**
   - テストデータ投入は最小限に
   - 削除する場合は必ず Firebase Console で個別確認

3. **顧客マスター(customers_temp)の一括削除**
   - 1,728件の顧客個人情報が消失する
   - 復旧はバックアップから手動

4. **セキュリティルールの無効化**
   - 現在「公開状態」だが、応急処置で2027/6/1まで限定的に許可
   - これを「全期間 if true」等に変更すると永続的にデータが第三者に晒される

5. **GitHub Web Editor で既存の閉じタグ・閉じ括弧を消す**(2026-05-20 追記)
   - `</section>`、`</script>`、`}`、`)`、`,` を不用意に削除しない
   - 詳細は「12. GitHub Web Editor 編集時の注意事項」参照

### ⚠️ 控えること

1. **CSV大量取込を1日複数回**
   - クォータを大きく消費する
   - 1日1回まで、できれば閉店後に実施

2. **Firebase Console を開きっぱなし**
   - 確認時だけ開いて、すぐ閉じる
   - 開きっぱなしだと使用状況自動更新で無駄な接続が発生

3. **日向さんが customers タブ・サブスクタブを操作**
   - 役割分担違反
   - 新規登録が必要な場合は Tak さんに依頼

4. **長文(50行以上)を1回でペースト**(2026-05-20 追記)
   - GitHub Web Editor は途中でコードが切れることがある
   - 詳細は「12. GitHub Web Editor 編集時の注意事項」参照

## 9. 既知の制約

### 9.1 customers_temp のリアルタイム性放棄

- 別端末・別タブで顧客が追加されても、即時反映されない(最大24時間遅延)
- 設計上の意図: クォータ削減のため customers_temp のみ onSnapshot を使わずキャッシュ化
- 影響: 日向さん端末で新規顧客が見えない場合あり
- 対処: 役割分担で日向さんは新規登録しない運用にしている

### 9.2 セキュリティルールが「公開状態」

- 現状: `allow read, write: if request.time < timestamp.date(2027, 6, 1);`
- 意味: 2027/6/1までは誰でもFirestoreにアクセス可能
- 個人情報保護法上の懸念あり
- 対処: 5月下旬〜6月初に Firebase 匿名認証を本格実装予定(宿題リスト参照)

### 9.3 Alpine Expression Error(既存バグ)

- ブラウザConsoleに `testDataFound.length` の null エラーが多数表示される
- 機能への影響なし
- 対処: いつか暇な時に `testDataFound: []` の初期化追加で修正予定

### 9.4 GitHub editor の落とし穴

- ペースト時に `</script>` タグを削除してしまうケースあり → `SyntaxError: Unexpected token '<'` 発生
- 長文ペースト時、無音でコードの後半が切れる事案あり(2026-05-19、約120行のうち約20行が欠落)
- GAS deploy URL を browser/GAS UI からコピーすると全角クォート混入する事例あり
- GitHub Pages のキャッシュは Shift + リロードまたはシークレットモードで確認
- **詳細対策: 「12. GitHub Web Editor 編集時の注意事項」参照(2026-05-20 追記)**

### 9.5 localStorage 容量

- 現状: 1,728件 × 平均1KB ≈ 1.7MB
- ブラウザ上限: 5〜10MB(余裕あり)
- 顧客が大幅に増えた場合(例: 5,000件超)は IndexedDB への移行検討

### 9.6 サブスク機能の暫定運用(v0.9系、〜2026-05-31)

- スクエア契約は来院記録に「サブスク初回(契約日決済)」として記録(月次払い込み記録は使わない)
- ライフカード契約は来院毎に「サブスク利用」を来院記録に記入
- 6/1から subscription_payments による本番運用開始予定

## 12. GitHub Web Editor 編集時の注意事項(2026-05-20 追記)

GitHub Web Editor で index.html を直接編集する際の事故防止手順。

### 過去の事故記録

#### 事件1: 5/18 真っ白事件(④-A コミット2 第1回)
- **症状**: コミット直後にアプリが真っ白(`Uncaught SyntaxError`)
- **原因**: HTML編集時、既存の `</section>` を誤って削除
- **対処**: 1コミット前にrevert → 再挑戦

#### 事件2: 5/19 SyntaxError事件(④-A コミット2 第2回)
- **症状**: コミット直後にアプリが真っ白(`Uncaught SyntaxError: missing ) after argument list at :6208:3`)
- **原因**: GitHub Web Editor で長いJavaScriptコード(約120行)をペーストした際、後半約20行が無音で欠落
  - 切れた位置: `status: 'pending',` の直後
  - 欠落範囲: paymentMethod/note/createdAt/updatedAt の各フィールド、batch.set/forEach の閉じ括弧、await batch.commit() 以降、try/catch/finally、関数終端、終了コメント
- **対処**: 欠落箇所を特定 → 不足コードを再挿入 → 再commit

### 根本原因の共通点

両事件とも GitHub Web Editor の**長文ペースト時の不安定性**が主因:
- HTML編集の終端マーカー(`</section>`、`</div>`)が消える
- JavaScript の途中で切れる(特に約100行以上の長いブロック)
- Preview Diff では一見正常に見えるため、検出が遅れる

### 標準対策(必須手順化)

#### 1. ペースト前の準備
- ペースト先の**カーソル位置を明確に**:Enter で空行を作っておき、その空行にペースト
- ペースト**直前**に、ペーストする文字数(行数)を頭に入れておく(例:「これから120行貼る」)

#### 2. ペースト直後の検証(必須・3点セット)

**A. 終端マーカーの存在確認**(検索バーで以下を順に確認)
- ペーストした最後のキーワード(例:`/v0.9.0-a2 月次払い込み記録`)
- ペーストした関数の主要処理キーワード(例:`batch.commit`)
- 既存の次セクション開始マーカー(例:`/v0.9 サブスクヘルパー関数`)

**B. Preview Diff の最下部確認**
- 単に「Preview Diff を確認」ではなく、**最下部まで必ずスクロール**
- 既存の `</section>` や `// ===== /xxx =====` が**まだ存在している**ことを確認
- 行番号の連続性を確認(挿入後の行番号 = 挿入前の行番号 + 追加行数 になっているか)

**C. キーワード検索による「最後の追加行」存在確認**
- 差分パッチで指定した「最後の行」のキーワードが**ヒットすること**を確認
- ヒットしなければ後半欠落の決定的証拠

#### 3. コミット前の最終確認

ペースト後・コミット前に以下を1分かけて確認:
- [ ] 終端マーカーA、B、Cがすべて存在する
- [ ] 「終了コメント」(例:`// ===== /v0.9.0-a2 月次払い込み記録 =====`)が存在する
- [ ] 既存の次セクション開始(例:`// ===== /v0.9 サブスクヘルパー関数 =====`)が**そのまま残っている**(消えていない)
- [ ] 関数の末尾 `},` のカンマが消えていない

#### 4. コミット後の即時動作確認(最重要)

コミット直後、GitHub Pages反映を1〜2分待ってから:
- [ ] ハードリロード(`Cmd + Shift + R`)
- [ ] **画面が真っ白でないこと**(最優先確認項目)
- [ ] ヘッダーのバージョン表示が更新されていること
- [ ] DevTools Console に**赤いSyntaxError**が出ていないこと
- [ ] ロック解除でき、各タブに移動できること

**真っ白だった場合の即対処**:
- 慌てず、GitHub の Commit History から**1つ前のコミットにrevert**
- Takさんの業務には影響なし(Firestore のデータは安全)
- その後、エラー位置(`xxx.html:N:M` の N が行番号)を確認して、該当行付近の構造を確認

### 修復作業時の探索手順(再発時の検索順序)

GitHub Web Editor の検索バーで以下を順に試して、欠落箇所を特定:

1. **挿入したブロックの末尾コメント**を検索(例:`/v0.9.0-a2`)
   - ヒットしない → 後半欠落確定 → 4へ
   - ヒット → 5へ(別の問題)

2. **関数本体の主要キーワード**を検索(例:`batch.commit`、`writeBatch(db)`)
   - 段階的に検索することで「どこまでコードが入っているか」が分かる

3. **挿入したブロックの開始キーワード**を検索(例:`v0.9.0-a2: 月次払い込み記録`)
   - ヒットすれば挿入開始位置は正常
   - ヒットしなければそもそも何も挿入されていない

4. **欠落範囲の特定** → 差分パッチの欠落部分のみを再挿入

5. それ以外の構造的問題(`{`、`(`、`,` の不整合)→ エラー位置の前後を慎重に読む

### 教訓のまとめ

| パターン | 主因 | 検出方法 | 防止策 |
|---|---|---|---|
| HTML終端消失 | `</section>` 等の消去 | Preview Diff の連続性 | 終端を絶対に消さない、空行を間に作る |
| JS後半欠落 | ペースト途中で切断 | 終端コメントの存在検索 | ペースト後の3点セット検証 |
| カンマ漏れ | メソッド末尾 `,` 消失 | Preview Diff の境界確認 | カンマ位置を意識してペースト |

### 1パッチ = 1コミット = 動作確認 の原則

- **1つの大きな変更**を1コミットでやらない
- 構造変更(import追加、データ宣言)と UI追加、関数追加は**別コミット**にする
- 各コミット後に**必ず動作確認**してから次に進む
- 真っ白になっても**1コミットだけrevert**で済む

## 11.2 長文ペースト後半欠落事故 #2(2026-05-20 発生)

### 概要

5/19-20 のサブスク機能追加(④-A)に続き、`saveSubscription` writeBatch 化作業(④-B コミット3a)で同じ事故が再発。これにより、**GitHub Web Editor の長文ペースト後半欠落は確定的な癖**として扱うべきパターンであることが確認された。

- **発生**: ペースト直後、表示上は欠落に気づかず、コミット → ハードリロード → DevTools で SyntaxError 発覚
- **欠落範囲**: ペースト範囲の **後半約25行が無音で消失**

  - localStorage キャッシュ経由で部分表示は可能だったが、操作不可

### 復元手順(実践済み)

1. **revert は最終手段とする**
2. DevTools の SyntaxError の行番号(`tak-kaikei-v2/:5455:5`)を起点に、GitHub の `blob/main/index.html#L5440-L5485` を直接確認

6. 動作確認 A〜D を全て実施 → ALL PASS

**所要時間**: 原因特定からエラー消滅まで **約10分**

### 必須予防手順(次回以降の事故防止)

**【ペースト前】**
1. ペースト範囲の **末尾の特徴的なキーワード** を覚えておく
   - 例: メソッド差し替えなら、メソッドの閉じ `},` の前にある最後の処理(`this.saving = false;` など)
   - 例: メソッド差し替えなら、**次に来るべきメソッド名**(`async deleteSubscription(s) {` など)
2. ペースト範囲が **50行を超える場合**は、最初から2分割でのペーストを検討

**【ペースト直後】**
3. 検索バーで上記の**両方の**キーワードがヒットすることを確認
4. Preview Diff を最下部までスクロールして、**次の関数が正しく続いている**ことを目視確認
5. ペースト範囲の途中で `setDoc` などの古いキーワードが残っていないかも確認

**【コミット前】**
6. ヒットしなかったら **即やり直し**(Cancel changes → 再ペースト or 2分割ペースト)
7. 50行を超えるペーストで一発成功させるより、2分割の方が安全と認識する

**【コミット後】**
8. **ハードリロード(Cmd + Shift + R)が最優先確認手順**
9. DevTools Console を開き、`Uncaught SyntaxError` が出ていないことを確認
10. ロック画面が正常に表示されることを確認(真っ白でないこと)

### 11.DD サブスク契約者の来院記録入力ルール(2026-05-22 「サブスク(ライフ)」プリセット追加)
